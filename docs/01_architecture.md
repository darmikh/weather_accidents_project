# Архитектура проекта

## Компоненты системы

### 1. Источники данных
- **Wikipedia** - список городов России (парсинг)
- **Яндекс.Геокодер / Nominatim** - координаты городов
- **Open-Meteo API** - почасовые погодные данные (2014-2025)
- **ГИБДД API** - данные о ДТП (с 2015) и коды ОКАТО
- **DaData API** - коды ОКАТО для регионов (вспомогательный источник)

### 2. ETL-скрипты (Python)
Все скрипты находятся в `src/etl/`:
- `main.py` - оркестратор, запускает все этапы
- `cities_etl.py` - парсинг городов и получение координат
- `gibdd_okato_etl.py` - получение кодов ГИБДД
- `weather_etl.py` - загрузка погоды
- `gibdd_dtp_etl.py` - загрузка ДТП
- `refresh_datamart.py` - обновление витрины
- `database.py` - работа с Supabase API
- `logger_config.py` - настройка логирования
- `config.py` - конфигурация проекта

### 3. Хранилище (Supabase PostgreSQL)
**Основные таблицы:**
- `raw_cities` - сырые данные из Wikipedia
- `cities` - обработанные города с координатами
- `raw_weather_data` - сырые JSON от Open-Meteo
- `weather_hourly` - почасовые погодные данные
- `dtp_main`, `dtp_road_conditions`, `dtp_vehicles`, `dtp_participants` - данные о ДТП
- `dtp_load_log`, `dtp_retry_queue` - служебные таблицы
- `refresh_log` - лог обновлений

**Витрина данных:**
- `mv_dtp_analytics`- материализованное представление для дашборда

### 4. Визуализация (Yandex DataLens)
Дашборд состоит из четырех вкладок:
- **Общая статистика и тренды** - ключевые метрики, динамика, временные паттерны
- **Территориальные значения** - карта, таблица, проверка гипотез
- **Погода и дорожные условия** - анализ влияния погоды, освещения, недостатки дороги
- **О дашборде** - методология и дата обновления

### 5. Автоматизация (GitHub Actions)
- Ежедневный запуск ETL в 6:00 МСК
- Запись даты обновления в `refresh_log`
- Отображение даты на дашборде
